# 05 — Routing & Pages (TanStack Start → Void Pages mode)

The biggest single change. Drives the file moves for almost every other
spec section.

## Conceptual mapping

| TanStack Start                                   | Void Pages mode                                                                                           |
| ------------------------------------------------ | --------------------------------------------------------------------------------------------------------- |
| `src/routes/*.tsx` (file-based)                  | `pages/*.tsx` (file-based, same naming rules but `[id]` not `$id`)                                        |
| `src/routes/__root.tsx`                          | `pages/layout.tsx` (and `pages/_layouts/*.tsx` for named layouts if needed)                               |
| `src/routes/api/**/*.ts`                         | `routes/api/**/*.ts` (Hono `defineHandler` with `GET`/`POST`/`HEAD`/`PUT`/`DELETE` named exports)         |
| `createFileRoute(...)` + `Route.useLoaderData()` | `.server.ts` `loader` export + page receives props via `Props = InferProps<typeof loader>`                |
| `createServerFn().handler(...)` mutations        | `.server.ts` `action` (or `actions` for multi-method pages) + client calls `useForm.post()` or `action()` |
| `createServerOnlyFn(...)`                        | Either an `action` (if user-initiated) or a regular Hono `defineHandler` in `routes/`                     |
| `beforeLoad` (route-level)                       | Inline `requireAuth(c)` / status check in the loader; redirect via `c.redirect(...)`                      |
| `Route.useRouteContext()`                        | `useShared()` (middleware-set) or page props (loader-set)                                                 |
| `Link` from `@tanstack/react-router`             | `Link` from `@void/react`                                                                                 |
| `useNavigate()` / `redirect({ to })`             | `useRouter().visit(...)` from `@void/react` / `c.redirect(...)` server-side                               |
| `validateSearch` (Zod-based)                     | Read `c.req.query()` in loader; validate with `withValidator({ query: … })`                               |
| `viewTransition` prop on `Link`                  | Built-in via Void Router scroll restoration; for explicit view transitions see Void pages-routing docs    |
| Generated `routeTree.gen.ts`                     | Generated by `@void/react` / `voidPlugin()` into `.void/*.d.ts` — do not commit                           |
| `useSuspenseQuery(getRecipeListOptions())`       | Direct prop access on the page component (loader returns the data)                                        |

## Filename rules

Convert:

- `$id.tsx` / `$id.ts` → `[id].tsx` / `[id].ts`
- `edit.$id.tsx` → `edit/[id].tsx` (Void uses directory nesting for nested params; flat dotted form for HTTP methods)

`pages/_layouts/` is **reserved** for named layouts. Don't put pages in there.

Files starting with `_` are ignored by the route scanner.

## `pages/layout.tsx` — root layout

Replaces `__root.tsx`. The current `__root.tsx`:

- Sets up theme via `getTheme()` in `beforeLoad`
- Reads `authUser` via `getAuthUser`
- Registers a service worker in `useEffect`
- Renders `<html>/<head>/<body>`, `<ToastProvider>`, `<Navbar>`, `<Outlet>`,
  TanStack devtools

Target shape:

```tsx
// pages/layout.tsx
import { Link, useShared } from '@void/react'
import OfflineBanner from '@/components/error/offline-banner'
import { Navbar } from '@/components/navigation/navbar'
import { ToastProvider } from '@/components/ui/toast'
import { ServiceWorker } from '@/components/service-worker' // extracted from __root useEffect

export default function Layout({ children }: { children: React.ReactNode }) {
  const { auth, theme } = useShared()
  return (
    <html className={theme} lang="fr">
      <head>{/* head tags */}</head>
      <body className="…">
        <ToastProvider>
          <OfflineBanner />
          <header className="…">
            <Navbar />
          </header>
          <main className="…">{children}</main>
        </ToastProvider>
        <ServiceWorker />
      </body>
    </html>
  )
}
```

Theme retrieval (`getTheme()` reads a cookie today) moves into a
middleware that sets `shared.theme`:

```ts
// middleware/01.theme.ts
import { defineMiddleware } from 'void'
import { getCookie } from 'hono/cookie' // npm install hono

declare module 'void' {
  interface CloudContextVariables {
    shared: { theme: 'light' | 'dark'; auth: /* … from 04-auth */ }
  }
}

export default defineMiddleware(async (c, next) => {
  const theme = (getCookie(c, 'theme') as 'light' | 'dark' | undefined) ?? 'light'
  // merge with existing shared.auth set by middleware/02.account-status.ts
  c.set('shared', { ...(c.get('shared') ?? {}), theme })
  await next()
})
```

The `<head>` configuration (links, meta, title) moves to either:

- `void.json` `head.*` (per `node_modules/void/skills/void/docs/reference/config.md`), or
- a per-page `head()` export (Void `head.md` docs).

The current `head` is shared across all pages, so prefer `void.json`:

```json
{
  "head": {
    "title": "Recipe Organizer",
    "htmlAttrs": { "lang": "fr" },
    "meta": [
      { "charset": "utf-8" },
      {
        "name": "viewport",
        "content": "width=device-width, initial-scale=1, user-scalable=no, viewport-fit=cover, interactive-widget=resizes-content"
      },
      { "name": "theme-color", "content": "#2f0d68" }
    ],
    "link": [
      { "rel": "manifest", "href": "/manifest.json" },
      { "rel": "icon", "href": "/favicon.ico" },
      { "rel": "apple-touch-icon", "href": "/apple-touch-icon.png" }
    ]
  }
}
```

The CSS import (`import appCss from '../styles/app.css?url'`) becomes a
direct import in `pages/layout.tsx`: `import '@/styles/app.css'` — Void
Pages mode handles asset injection automatically.

## Per-route migration notes

### `/` — Recipe list

```ts
// pages/index.server.ts
import { defineHandler, type InferProps } from 'void'
import { db } from 'void/db'
import { recipe } from '@schema'

export type Props = InferProps<typeof loader>

export const loader = defineHandler(async (c) => {
  return { recipes: await db.select().from(recipe) }
})
```

```tsx
// pages/index.tsx
import { Link } from '@void/react'
import type { Props } from './index.server'
import { useShared } from '@void/react'

export default function RecipeList({ recipes }: Props) {
  const { auth } = useShared()
  return (
    <ScreenLayout title="Recettes" pageKey="/">
      <div className="…">
        {recipes.map((r) => (
          <RecipeCard recipe={r} key={r.id} />
        ))}
      </div>
      {auth?.user && (
        <Button render={<Link href="/recipe/new">…</Link>} size="icon-xl">
          …
        </Button>
      )}
    </ScreenLayout>
  )
}
```

Cache-Control headers (currently set via `headers()` on the route) move to
`void.json` `routing.headers` if static, or to a `head()` export — but
HTTP cache headers belong server-side. Use the response wrapper in the
loader if needed, or `routing.revalidate` for ISR.

### `/recipe/[id]` — Recipe detail

Currently uses `getRecipeOptions({ id })` with `useSuspenseQuery`. Becomes:

```ts
// pages/recipe/[id].server.ts
export const loader = defineHandler(async (c) => {
  const id = Number(c.req.param('id'))
  const recipeRow = await db.query.recipe.findFirst({
    where: { id },
    with: { ingredientGroups: { with: { groupIngredients: { with: { ingredient: true } } } }, linkedRecipes: true },
  })
  if (!recipeRow) return c.notFound()
  return { recipe: recipeRow }
})

export const action = defineHandler(async (c) => {
  // delete-recipe action
})
```

The component uses props directly. `LinkedRecipesContext` (currently a
React context in `src/features/recipe/contexts/linked-recipes-context.tsx`)
can either stay client-side or move into the loader return if all data is
needed up front.

### `/search` — Recipe search

The current implementation parses `?q=` via `validateSearch` + Zod. New:

```ts
// pages/search.server.ts
import * as v from 'valibot'

export const loader = defineHandler.withValidator({
  query: v.object({ q: v.optional(v.string()) }),
})(async (c, { query }) => {
  const term = query.q?.trim() ?? ''
  return { results: await searchRecipes(term) }
})
```

Or stay on Zod via `void/drizzle-zod` Standard Schema interop.

### `/settings/users` — multi-action page

Today multiple server functions (`approve`, `block`, `create`, `getAll`).
Becomes a single page with `actions`:

```ts
// pages/settings/users.server.ts
export const loader = defineHandler(async (c) => {
  const me = requireAuth(c)
  if (me.role !== 'admin') return c.redirect('/')
  return { users: await db.select().from(user).orderBy(asc(user.email)) }
})

export const actions = {
  approve: defineHandler.withValidator({ body: v.object({ id: v.string() }) })(/* … */),
  block: defineHandler.withValidator({ body: v.object({ id: v.string() }) })(/* … */),
  create: defineHandler.withValidator({ body: v.object({ email: v.string() }) })(/* … */),
}
```

UI invokes them via `useForm('/settings/users?approve', …)` or
`action('/settings/users?block', { data: { id } })`.

### `/shopping-list` — client-only

No `.server.ts` companion. The page is a pure client view over the Zustand
store. Render-time it can read nothing from the server.

If SEO crawlers should not see an empty shell, add a tiny `head()` export
with `noindex`.

## Loaders, deferred props, ISR

- Most loaders are sub-100ms (D1 read) and don't need `defer(...)`.
- The recipe list and search pages might benefit from
  `routing.revalidate` ISR (today they use HTTP `Cache-Control`). Defer
  the decision until perf data is in; ISR can be added incrementally.

## Type-shared `Props`

Each `.server.ts` exports `export type Props = InferProps<typeof loader>`.
The component imports it. **Do not** re-declare `Props` interfaces by hand.

## `useShared()` typing

Augment `CloudContextVariables` exactly once (in `middleware/02.account-status.ts`
or in `src/types/shared.ts`) with the full `shared` shape. Then `useShared()`
returns the typed value across the app.

- `src/router.tsx`

## Verification gate

This phase is "done" when:

- Every URL in the [Route inventory](#route-inventory) renders under
  `vp dev` via the new `pages/`+`.server.ts` files
- Browser dev tools show no console errors or hydration mismatches on
  any route
- Navigation between routes uses `@void/react` `Link` and doesn't fall
  back to full-page reloads
- Loader props flow through to components without intermediate fetches
- All `routes/api/**` endpoints respond correctly (image, video, auth)
