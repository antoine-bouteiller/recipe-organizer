# Auth cutover scripts

Three-step SQL choreography for migrating the existing 3 production users
from the custom `user` table to Better Auth's `user` + `account` tables
without breaking the logical reference from `recipe.created_by`.

## Why this is needed

- The legacy `user` table and Better Auth's `user` table share the same
  name. Better Auth's schema migration would fail (or silently override)
  if the legacy table is still in place.
- The legacy `user.id` is the user's **Google OAuth sub** (provider
  account ID). It's the same value `recipe.created_by` points at. We
  must preserve those IDs verbatim so the existing recipes stay tied to
  their owners.
- There is **no enforced foreign key** on `recipe.created_by ŌåÆ
user.id` (verified by grep against `migrations/*.sql`). The reference
  is logical only, so we don't need to disable FK enforcement during
  the swap ŌĆö but we still must preserve the IDs.

## Run order

1. **`01_rename_legacy_user.sql`** ŌĆö run before Better Auth's schema
   migration. Renames the existing `user` table to `legacy_user` to make
   room.

2. **Better Auth schema migration** ŌĆö generated by
   `void db generate` once `auth.providers: ["google"]` is in `void.json`
   and the additionalFields are in `auth.ts`. This emits a normal
   `db/migrations/<ts>_<name>.sql` that creates `user`, `account`,
   `session`, `verification` (with `status` + `role` columns from D15).
   Apply it via `wrangler d1 migrations apply DB --remote`.

3. **`02_backfill_users_from_legacy.sql`** ŌĆö run after step 2.
   - Copies each row from `legacy_user` to `user` (preserving `id`).
   - Inserts a matching `account` row with `provider: 'google'` and
     `accountId = id`.
   - Drops `legacy_user`.

## How to apply these

Both files are intended to be added as **regular Drizzle/Void migrations**
under `db/migrations/`. Copy with appropriate `<timestamp>_<name>.sql`
names so they land in the right position in the lineage:

```bash
TS=$(date -u +%Y%m%d%H%M%S)
cp void-migration/scripts/01_rename_legacy_user.sql \
   "db/migrations/${TS}_rename_legacy_user.sql"
# ŌĆ” run void db generate to emit the Better Auth schema migration ŌĆ”
TS=$(date -u +%Y%m%d%H%M%S)
cp void-migration/scripts/02_backfill_users_from_legacy.sql \
   "db/migrations/${TS}_backfill_users_from_legacy.sql"
```

Then apply locally first:

```bash
void db reset       # wipes local D1, replays full migration set including
                    # the two cutover scripts
```

After local verification, apply to production:

```bash
wrangler d1 export recipe-organizer --remote \
  --output ./pre-auth-cutover.sql      # snapshot first
wrangler d1 migrations apply DB --remote
```

## Verification checklist

After step 3 on production:

- `SELECT count(*) FROM user;` returns 3.
- `SELECT count(*) FROM account WHERE providerId='google';` returns 3.
- `SELECT count(*) FROM legacy_user;` errors with "no such table".
- Each of the 3 users can sign in with Google.
- `SELECT count(*) FROM recipes WHERE created_by NOT IN (SELECT id FROM user);`
  returns **0** ŌĆö no orphaned recipes.

If the verification fails, restore from `./pre-auth-cutover.sql`.

## Schema assumption

Better Auth's `account` table is assumed to have these required fields
(all others nullable for social providers): `id`, `userId`, `providerId`,
`accountId`, `createdAt`, `updatedAt`. Confirm against the migration
emitted by `void db generate` ŌĆö if the generated schema requires
additional NOT NULL columns, add them to the INSERT in
`02_backfill_users_from_legacy.sql` before running.
